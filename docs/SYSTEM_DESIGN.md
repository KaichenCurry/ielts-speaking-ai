> 本文档面向面试官和潜在合作者，详细说明系统设计思路、产品思考和技术实现。

## 一、项目背景与产品定位

### 1.1 产品定位

**目标用户：雅思口语教师**

这是一个**面向教师的 AI 助教系统**，不是学生直接使用的 App。老师通过 Telegram Bot 布置作业、管理题库、查看周报；学生在 Bot 中完成练习并收到反馈。系统自动完成评分、存档工作，让老师从重复性工作中解放。

### 1.2 解决的问题

| 老师痛点 | 我们的解决方案 | 价值 |
|---------|-------------|------|
| 重复性评分工作繁重 | AI 自动评分 | 减少 80%+ 重复工作 |
| 作业反馈慢，学生错失记忆窗口 | 答题后即时逐句反馈 | 学习效果大幅提升 |
| 学生数据散落，难以追踪 | Notion 存档 | 永久留存，随时可查 |
| 难以批量管理学生进度 | 周报自动推送 | 周五早上全景一览 |
| 题库更新耗精力 | 每周自动更新 | 无需手动维护 |

### 1.3 目标用户分析

| 用户 | 需求 | 我们的方案 |
|------|------|-----------|
| 雅思口语老师 | 高效布置作业 | `/题目 Test XX` 一键发送 |
| 雅思口语老师 | 查看学生表现 | Notion 作业库随时调取 |
| 雅思口语老师 | 掌握班级整体情况 | 周五自动推送周报 |
| 雅思口语老师 | 管理题库 | 每周三、六自动更新 |

## 三、核心功能详解

### 3.1 作业管理模块

#### 老师操作流程

```
老师发送命令：
/题目 Test 07

    ↓
Bot 解析并发送：
├── Part 1（5题）
├── Part 2（1题 Cue Card）
└── Part 3（5题）

    ↓
学生在家随时答题
```

#### 题库管理

| 功能 | 说明 |
|------|------|
| 题库规模 | 66 套真题 |
| 话题覆盖 | 人物/地点/物品/事件/活动/习惯/食物全类别 |
| 题库自动更新 | 每周三、六凌晨自动更新 |
| 题目格式 | Part 1（最多5题/topic）、Part 2（Cue Card + 4提示点）、Part 3（最多5题） |

#### 话题体系

| 类别 | 示例话题 |
|------|---------|
| 人物类 | 制定计划的人、想见的名人、影响我的老师 |
| 地点类 | 博物馆、历史景点、花园/公园、靠近水的地方 |
| 物品类 | 收藏品、礼物、衣服、一张照片、DIY项目 |
| 事件类 | 难忘旅行、与朋友做的事、出席婚礼、童年记忆 |
| 活动类 | 运动、电影/剧集、音乐、游戏、网站 |
| 习惯类 | 健康习惯、晨间 routine、学到的技能 |
| 食物类 | 喜欢的餐厅、喜欢的水果、热饮 |

### 3.2 AI 评测模块

#### 多模型协同架构

```
学生语音
    ↓
Whisper（语音 → 文字）
    ↓
RAG 检索（Notion 历史错题本 + 作业库）
    ↓
MiniMax 大语言模型（5维度评分）
    ↓
结构化输出（Band Score + 逐句反馈）
    ↓
Notion 存档
```

#### 评分维度

| 维度 | 关注点 | 示例问题 |
|------|--------|---------|
| 语法 | 主谓一致、从句使用、介词搭配 | "He go" → "He goes" |
| 词汇 | Chinglish、同义词替换、高分词汇 | "很贵" → "expensive" / "costly" |
| 时态 | 过去时、现在时、完成时、混合时态 | 过去经历用现在时 |
| 逻辑 | 因果关系、转折、层次感、跑题 | 观点与举例不匹配 |
| 思路 | 举例是否具体、论证深度、独特观点 | 举例泛泛而谈 |

#### Band Score 计算

```
综合 Band = Part1×30% + (Part2×40% + Part3×60%)×70%
```

**计算示例**：
- Part 1 均分：6.0
- Part 2 得分：6.5
- Part 3 均分：6.0

```
Part2_3合成 = 6.5×0.4 + 6.0×0.6 = 2.6 + 3.6 = 6.2
综合 Band = 6.0×0.3 + 6.2×0.7 = 1.8 + 4.34 = 6.14 ≈ 6.0
```

### 3.3 数据存档模块

#### Notion 数据库

| 数据库 | 用途 | 内容 |
|--------|------|------|
| 题库 | 66套真题 | 题目 + 话题分类 + 难度 |
| 作业反馈库 | 学生练习存档 | 答题原文 + Band Score + 逐句反馈 |
| 错题本 | 错误案例积累 | 错误类型 + 老师纠正 + 最终 Band |

#### 数据价值

- **学生端**：可查看历史成绩曲线，了解进步轨迹
- **老师端**：可按学生/按话题/按时间筛选，制定针对性教学计划
- **系统端**：RAG 检索来源，持续提升 AI 评分准确性

### 3.4 错题本模块

#### 老师纠正流程

```
老师发现学生错误 → 发送 /纠正 [正确表达]
    ↓
Bot 自动收录到错题本
    ↓
包含内容：
• 学生错误答案
• AI 原评判
• 老师正确纠正
• 错误类型分类（语法/词汇/时态/逻辑/思路）
```

#### 错题本价值

| 价值维度 | 说明 |
|---------|------|
| 教学参考 | 老师可查看班级常见错误，针对性备课 |
| RAG 来源 | 历史错题增强 AI 评分上下文 |
| 微调数据 | 积累 100+ 条后可启动模型微调 |

### 3.5 周报分析模块

#### 周报推送时间

每周五 18:00（美国东部时间）自动推送到 Telegram 群

#### 周报内容

```
📊 班级周报 | 2026.04.11-04.15

【本周练习概览】
• 练习人次：12
• 平均 Band：6.2
• 较上周变化：+0.3 ↑

【Band 分布】
• 7.0+：3人
• 6.0-6.5：6人
• 5.5-6.0：2人
• 5.0 以下：1人

【常见错误 TOP5】
1. 词汇：时态混用（过去时 vs 现在时）—— 出现 8 次
2. 语法：主谓不一致 —— 出现 6 次
3. 逻辑：举例与观点不匹配 —— 出现 5 次
4. 词汇：Chinglish 表达 —— 出现 4 次
5. 思路：论证深度不足 —— 出现 3 次

【下周教学建议】
• 重点关注时态一致性（可布置过去时话题练习）
• 加强举例具体化训练
• 建议学生背诵高分词汇替换表
```

#### 周报价值

| 用户 | 价值 |
|------|------|
| 老师 | 周五早上 2 分钟看完班级全景，无需手动统计 |
| 学生 | 了解班级整体水平，找到参照 |
| 系统 | 量化产品效果，指导迭代方向 |

## 四、题库自动更新机制

### 4.1 更新策略

| 更新频率 | 时间 | 内容 |
|---------|------|------|
| 自动更新 | 每周三、六 凌晨 02:00 | 新题 + 话题扩充 |
| 手动更新 | 随时 | `/更新题库` 指令 |

### 4.2 题库话题去重

**问题**：之前 66 个 Test 只用了 10 个循环话题，每话题重复 6-7 次

**解决**：预定义 66 个独特 IELTS 话题，确保每个话题只出现一次

**话题去重工具**：`update_unique.py`

### 4.3 话题分类体系

| 大类 | 话题数 | 覆盖内容 |
|------|--------|---------|
| 人物类 | 10+ | 家人、朋友、名人、老师等 |
| 地点类 | 10+ | 城市、景点、餐厅、公园等 |
| 物品类 | 10+ | 礼物、收藏品、衣服、照片等 |
| 事件类 | 15+ | 旅行、婚礼、童年、冒险等 |
| 活动类 | 10+ | 运动、电影、音乐、游戏等 |
| 其他类 | 10+ | 习惯、食物、季节、节日等 |

## 五、状态机设计

### 5.1 三段式流程

```
IDLE → AWAITING_PART1 → AWAITING_PART2 → AWAITING_PART3 → DONE → 写Notion → 发群 → IDLE
```

### 5.2 状态流转图

```
                    ┌─────────────┐
                    │    IDLE     │ ← 等待老师布置题目
                    └──────┬──────┘
                           │
               老师发送 /题目 Test XX
                           │
                           ▼
                    ┌─────────────────┐
                    │ AWAITING_PART1  │
                    └────────┬────────┘
                             │
              学生发送语音（Part 1，每题）
                             │
              Whisper → RAG检索 → 评分（暂存）
                             │
                             ▼
                    ┌─────────────────┐
                    │ AWAITING_PART2  │
                    └────────┬────────┘
                             │
              学生发送语音（Part 2）
                             │
              Whisper → RAG检索 → 评分（暂存）
                             │
                             ▼
                    ┌─────────────────┐
                    │ AWAITING_PART3  │
                    └────────┬────────┘
                             │
              学生发送语音（Part 3，每题）
                             │
              Whisper → RAG检索 → 综合评分（暂存）
                             │
                             ▼
                      ┌──────────┐
                      │   DONE   │ → 写Notion → 发群里 → IDLE
                      └──────────┘

    ═══════════════════════════════════════
                    【老师纠正路径】
              ┌─────────────────────┐
              │   任意状态都可接收  │
              └──────────┬──────────┘
                         │
              解析纠正 → 写入错题本
                         │
                         ▼
                   回复老师确认
                   （不改变主流程状态）
```

### 5.3 为什么不用复杂框架

| 场景特点 | 框架需求 |
|---------|---------|
| 固定流程 | Part 1→Part 2→Part 3 三段式 |
| 状态简单 | IDLE/Part1/Part2/Part3/Done 五种状态 |
| 无分支逻辑 | 顺序执行即可 |
| 多渠道复用 | 状态机统一处理 |

**结论**：Python 字典或简单类就能实现状态管理，不需要 LangGraph/AutoGen 等框架。

## 六、AI 评测指标体系

### 6.1 核心指标

| 指标 | 计算方式 | 目标 | 实际 |
|------|---------|------|------|
| Band 误差 | AI Band - 老师纠正后 Band 的绝对值 | ≤0.3 | 0.2 ✅ |
| 维度准确率 | AI 标记的错误被老师确认的比例 | ≥85% | 达标 ✅ |
| 错题命中率 | AI 识别的错误中真正存在的比例 | ≥80% | 达标 ✅ |
| 格式正确率 | 评分输出符合格式要求的比例 | ≥98% | 98%+ ✅ |

### 6.2 评估触发机制

```
每周五 18:00 自动触发评估
    ↓
抽样本周 10 条作业
    ↓
对比 AI 评分 vs 老师纠正（如果有）
    ↓
计算各项指标
    ↓
超标 → 发预警给老师
```

## 七、Prompt 设计

### 7.1 评分 Prompt 结构

```
System Prompt（固定）
    ├── 角色定义：你是一个雅思口语个性化反馈助手
    ├── 5个维度定义（语法/词汇/时态/逻辑/思路）
    ├── 逐句分析原则
    ├── Band 计算规则
    └── 禁忌（不对照标准答案，只看学生说的）

User Prompt（动态）
    ├── 学生转写文本
    ├── RAG 检索结果（历史错题/同类回答）
    └── 具体问题（如果有）
```

### 7.2 Prompt 迭代记录

| 版本 | 日期 | 问题 | 修改 | 效果 |
|------|------|------|------|------|
| V2 | 2026-04-13 | 维度名称不统一 | 统一为5维度中文名称 | 格式一致性 78%→98% |
| V1 | 2026-04-11 | Band 系统性偏高 0.5 | 增加校准规则 | 误差从 0.5→0.2 |

## 八、数据飞轮

```
学生答题 → AI评分 → 老师纠正 → 错题本收录 → RAG增强 → 微调数据池
                                ↑
                         关键！这不是凭空生成的
                         是专家人工审核过的"正确答案"
```

### 什么时候需要微调？

**信号**（出现任意一个就考虑）：
- `/纠正` 频率高：每周>10条
- 同一错误重复出现
- AI 输出格式不稳定
- Band 系统性偏差

## 九、复盘与优化方向

### 9.1 做得好的

- 产品闭环完整，真正解决了教学痛点
- AI 能力设计克制，没有为了用技术而用技术
- 数据飞轮设计为后续能力进化留足空间

### 9.2 可以优化的

| 方向 | 当前状态 | 优化目标 |
|------|---------|---------|
| 学生主动提问 | 缺失 | 增加学生与 AI 助教对话的入口 |
| RAG 升级 | 轻量版 | 数据量上来后升级向量检索 |
| 微调启动 | 未启动 | 错题本积累到 100 条后验证效果 |
| 学生端 UI | 缺失 | 增加学生视角的数据展示 |
