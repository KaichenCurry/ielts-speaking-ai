# 雅思口语 AI 练习与评测系统 - 详细设计文档

> 本文档面向面试官和潜在合作者，详细说明系统设计思路、产品思考和技术实现。

---

## 一、项目背景与产品定位

### 1.1 解决的问题

雅思口语备考面临三大核心痛点：

| 痛点 | 现状 | 影响 |
|------|------|------|
| 师资稀缺 | 好老师供给不足，一对一陪练成本高（300-800元/小时） | 学习门槛高，练习机会少 |
| 反馈滞后 | 学生录音后需等待数小时甚至次日才能收到反馈 | 错失最佳记忆窗口，反馈效果大打折扣 |
| 反馈质量参差 | 人工评分主观性强，不同老师标准不一 | 学生难以建立稳定的进步感知 |

### 1.2 目标用户

| 用户 | 需求 |
|------|------|
| 雅思备考学生 | 低成本、高频次、即时反馈的练习机会 |
| 口语教师 | 从重复性评分工作中解放，专注教学干预 |

### 1.3 产品定位

**做"AI助教"而非"AI评分器"**。评分只是起点，真正的价值在于反馈——不是冷冰冰的分数，而是逐句告诉你哪里说得好、哪里需要改、怎么改。

## 二、产品架构

### 2.1 完整流程

```
老师发送 /题目 Test XX
    ↓
Telegram Bot 解析题目 + 锁定状态
    ↓
┌─────────────────────────────────────────┐
│ Part 1（5题循环）                        │
│ Bot 发题 → 学生语音 → Whisper 转写        │
│ → RAG 增强上下文 → MiniMax AI 评分暂存    │
│ → 追问下一题（不展示反馈）                │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Part 2（1题）                            │
│ Bot 发题卡 → 学生语音（1-2分钟）          │
│ → Whisper → RAG → MiniMax 评分暂存       │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Part 3（5题循环）                        │
│ 同 Part 1 流程                           │
└─────────────────────────────────────────┘
    ↓
全部完成 → 汇总评分 → 发群 + 写 Notion
```

### 2.2 状态机设计

```
IDLE → AWAITING_PART1 → AWAITING_PART2 → AWAITING_PART3 → DONE → 写Notion → 发群 → IDLE
```

**设计理由**：
- 三段式固定流程，状态简单清晰
- 支持老师中途发送 `/纠正` 打断流程
- 每个状态都有明确的入口动作和出口动作

### 2.3 为什么不用复杂框架（LangGraph/AutoGen）

| 场景特点 | 框架需求 |
|---------|---------|
| 固定流程 | Part 1→Part 2→Part 3 三段式 |
| 状态简单 | IDLE/Part1/Part2/Part3/Done 五种状态 |
| 无分支逻辑 | 顺序执行即可 |
| 多渠道复用 | 状态机统一处理 |

**结论**：Python 字典或简单类就能实现状态管理，不需要过度设计。

## 三、AI 能力设计

### 3.1 多模型协同架构

```
学生语音
    ↓
Whisper（语音 → 文字）
    ↓
RAG 检索（Notion 历史错题本 + 作业库）
    ↓
MiniMax 大语言模型（5维度评分）
    ↓
结构化输出（Band Score + 逐句反馈）
    ↓
Notion 存档
```

### 3.2 各环节设计理由

| 环节 | 技术选型 | 设计理由 |
|------|---------|---------|
| 语音识别 | Whisper | 低延迟，支持长时间录音，无需静音检测，口语场景效果优秀 |
| 评分推理 | MiniMax | 中文语境理解好，支持结构化输出，性价比高 |
| 上下文增强 | RAG（轻量版） | 历史错题复用，无需额外向量数据库，够用就好 |
| 数据存储 | Notion | 已有结构化数据，老师可随时查看，无需自建数据库 |

### 3.3 RAG 设计

#### 什么时候用 RAG？

| 时机 | 检索内容 | 用途 |
|------|---------|------|
| 布置题目时 | 该话题历史错题 | 给老师参考，提醒学生易错点 |
| 答题过程中 | 同类话题常见错误模式 | 增强评分上下文（静默，不输出） |
| 全部答完后汇总时 | Part 2 发现的问题 + Part 3 关联 | 生成综合评分总览 |
| 老师纠正时 | 该学生在该话题的历史表现 | 判断是否重复犯错 |

#### 为什么不用向量数据库？

| 方案 | 适合场景 | 本系统情况 |
|------|---------|-----------|
| Pinecone/Chroma | 大量非结构化文档检索 | 数据量还不够大（<1000条） |
| 关键词检索 | 结构化数据 + 固定字段 | ✅ Notion 数据已结构化 |
| 混合方案 | 以后数据量大时升级 | 预留升级空间 |

**结论**：当前用关键词检索足够。等错题本积累到 500+ 条时再考虑升级向量数据库。

## 四、评测指标体系

### 4.1 核心指标

| 指标 | 计算方式 | 目标 | 实际 |
|------|---------|------|------|
| Band 误差 | AI Band - 老师纠正后 Band 的绝对值 | ≤0.3 | 0.2 ✅ |
| 维度准确率 | AI 标记的错误被老师确认的比例 | ≥85% | 达标 ✅ |
| 错题命中率 | AI 识别的错误中真正存在的比例 | ≥80% | 达标 ✅ |
| 格式正确率 | 评分输出符合格式要求的比例 | ≥98% | 98%+ ✅ |

### 4.2 迭代机制

```
每周五 18:00 自动触发评估
    ↓
抽样本周 10 条作业
    ↓
对比 AI 评分 vs 老师纠正（如果有）
    ↓
计算各项指标
    ↓
超标 → 发预警给老师 → 分析原因 → 优化 Prompt
```

## 五、Prompt 设计

### 5.1 评分 Prompt 结构

```
System Prompt（固定）
    ├── 角色定义：你是一个雅思口语个性化反馈助手
    ├── 5个维度定义（语法/词汇/时态/逻辑/思路）
    ├── 逐句分析原则
    ├── Band 计算规则（Part1×30% + (Part2×40%+Part3×60%)×70%）
    └── 禁忌（不对照标准答案，只看学生说的）

User Prompt（动态）
    ├── 学生转写文本
    ├── RAG 检索结果（历史错题/同类回答）
    └── 具体问题（如果有）
```

### 5.2 逐句反馈格式

```
原句： "{学生原话}"

• 语法：❌/✅/⚠️ + 具体解释（原词 → 正确词）
• 词汇：❌/✅/⚠️ + 具体解释（原词 → 正确词）
• 时态：❌/✅/⚠️ + 具体解释
• 逻辑：❌/✅/⚠️ + 具体解释
```

### 5.3 Prompt 迭代记录

| 版本 | 日期 | 问题 | 修改 | 效果 |
|------|------|------|------|------|
| V2 | 2026-04-13 | 维度名称不统一，有时写"词汇"有时写"LR" | 统一为5维度中文名称 | 格式一致性 78%→98% |
| V1 | 2026-04-11 | Band Score 系统性偏高 0.5 分 | 增加校准规则——有明显错误必须扣分 | 误差从 0.5→0.2 |

## 六、数据飞轮

```
学生答题 → AI评分 → 老师纠正 → 错题本收录 → RAG增强 → 微调数据池
                                ↑
                         关键！这不是凭空生成的
                         是专家人工审核过的"正确答案"
```

### 什么时候需要微调？

**信号**（出现任意一个就考虑）：
- `/纠正` 频率高：每周>10条
- 同一错误重复出现：比如"语法错误总是漏掉"
- AI 输出格式不稳定：评分模板总是不统一
- Band 系统性偏差：总是偏高或偏低 0.5+

### 微调数据要求

| 检查项 | 要求 |
|--------|------|
| 数据量 | ≥100条（当前不足，先积累） |
| 来源 | 必须来自老师/纠正，不能纯AI生成 |
| 覆盖 | 5个维度都要有（语法/词汇/时态/逻辑/思路） |
| 分布 | 不同 Band 分数段都要有（4.5-8分） |
| 质量 | 老师确认过的评分结果 |

## 七、产品亮点

### 7.1 异步评分架构

录音后**立即发下一题**，后台并发评分，消除学生等待感。

设计思考：雅思口语考试要求学生连续说 1-2 分钟，如果等学生说完再评分再发下一题，会造成明显的等待间隙，体验不流畅。异步架构让流程如行云流水。

### 7.2 无标准答案设计

反馈以学生自己的答案为中心，不与参考答案对比。

设计思考：传统评分往往对照"标准答案"，但雅思口语没有标准答案，关键是表达是否准确、流畅、地道。以学生为中心的设计更符合口语评测本质。

### 7.3 反馈时机设计

答题过程中**不显示反馈**，全部答完后才展示。

设计思考：即时反馈会打断答题节奏，也会让学生过度关注单题表现而失去整体发挥。延迟展示让学生先完整表达，再统一收反馈，效果更好。

### 7.4 数据飞轮设计

从"错题本"到"RAG"到"微调"，系统可以随着数据积累持续进化。

设计思考：AI 评分系统不可能一步到位，需要持续迭代。数据飞轮让每一次老师纠正都变成系统的学习素材，越用越聪明。

## 八、复盘与优化方向

### 8.1 做得好的

- 产品闭环完整，从出题到反馈到存档，真正解决了教学痛点
- AI 能力设计克制，没有为了用技术而用技术（如选择轻量 RAG 而非一上来就用向量数据库）
- 数据飞轮设计为后续能力进化留足空间

### 8.2 可以优化的

| 方向 | 当前状态 | 优化目标 |
|------|---------|---------|
| 学生主动提问 | 缺失 | 增加学生与 AI 助教对话的入口 |
| RAG 升级 | 轻量版 | 数据量上来后升级向量检索 |
| 微调启动 | 未启动 | 错题本积累到 100 条后验证效果 |
| 学生端 UI | 缺失 | 增加学生视角的数据展示 |
