> 从 0 到 1 设计并落地雅思口语 AI 助教系统，打造「作业布置→AI评测→逐句反馈→Notion存档→周报分析」完整产品闭环，解决老师重复性评分工作繁重的问题。

**可量化版本**：
> 从 0 到 1 设计并落地雅思口语 AI 助教系统，实现老师重复性评分工作量减少 80%+，一条指令即可完成作业布置、评分、存档全流程。

---

### 2. RAG 检索增强架构
> 设计轻量级 RAG 方案，从 Notion 历史错题本和作业库中检索同类话题错误模式，为 AI 评分注入上下文，**评分准确性提升显著**。

**面试展开点**：
- 为什么不用向量数据库（数据量 < 1000 条、Notion 数据已结构化）
- RAG 的检索链路：错题本 + 作业库 → 关键词检索 → 上下文注入
- RAG 的价值：让 AI 评分有历史参考，不是凭空打分

---

### 3. 提示词设计与迭代优化
> 设计 5 维度评分 Prompt 框架（语法/词汇/时态/逻辑/思路），通过周级评估发现 Band 系统性偏高问题，**迭代 2 轮后评分误差从 0.5 收窄至 0.2**，格式正确率稳定在 98%+。

**面试展开点**：
- Prompt 设计：怎么定义 5 个维度、怎么设计逐句反馈格式
- 迭代方法：怎么发现 Band 偏差（周级评估）、怎么验证改进了（抽样对比）
- 迭代结果：误差 0.5→0.2 不是偶然，是数据驱动的优化过程

---

### 4. Agent Workflow 编排
> 设计 Part1→Part2→Part3 三段式状态机，实现异步评分架构（录音后立即发下一题、全部答完才展示反馈），**解决即时反馈与考试心态的两难**，体验流畅无等待。

**面试展开点**：
- 为什么需要状态机（学生在任意时刻打断时系统要保持正确状态）
- 异步架构的挑战（评分不能让学生看到、但又要即时追问下一题）
- 为什么不用 LangGraph（场景简单、Python 类就能实现，不需要过度设计）

---

### 5. 多模型协同与选型
> 设计 Whisper（语音识别）+ MiniMax（评分推理）的多模型协同方案，Whisper 处理口语转写、MiniMax 基于上下文进行 5 维度评分，**模型链路清晰、端到端自动化**。

**面试展开点**：
- 为什么选 Whisper（开源、口语场景效果好、可本地运行）
- 为什么选 MiniMax（中文理解强、性价比高、通过 OpenClaw 集成）
- 多模型怎么串联（语音→转写→RAG→评分→存档）

---

### 6. 模型效果评测体系
> 建立 AI 评测指标体系（Band 误差、维度准确率、错题命中率），每周五自动抽样 10 条作业对比 AI vs 老师评分，**量化 AI 质量并触发迭代**，实现数据驱动的模型优化闭环。

**面试展开点**：
- 指标怎么定义（Band 误差 = |AI Band - 老师确认 Band|）
- 评估怎么触发（每周五 18:00 定时运行）
- 迭代怎么闭环（超标 → 发预警 → 分析原因 → 优化 Prompt）

---

### 7. 数据飞轮与微调规划
> 设计「学生答题→AI评分→老师纠正→错题本收录→RAG增强→微调数据」数据闭环，每一次老师纠正都是专家审核的高质量标注，**为未来模型微调奠定数据基础**（当前错题本持续积累中）。

**面试展开点**：
- 数据飞轮的价值（越用越聪明）
- 什么时候启动微调（错题本 ≥ 100 条）
- 微调数据的要求（5 个维度全覆盖、不同 Band 分数段都有）

### 1. RAG 检索增强架构

**设计链路**：
```
学生答题 → Whisper 转写 → RAG 检索（历史错题 + 同类回答）→ MiniMax 评分
```

**为什么不用向量数据库**：
- 数据量 < 1000 条，关键词检索足够
- Notion 数据已结构化，不需要文本切分
- "够用就好"，不过度设计

**RAG 的价值**：
- 让 AI 评分有历史上下文参考
- 同类话题的错误模式能提升评分准确性
- 是数据飞轮的关键环节

### 2. 提示词设计与迭代优化

**Prompt 结构**：
```
System: 角色定义 + 5维度框架 + Band计算规则 + 禁忌
User: 学生转写文本 + RAG上下文
```

**迭代记录**：

| 版本 | 日期 | 问题 | 修改 | 效果 |
|------|------|------|------|------|
| V2 | 2026-04-13 | 维度名称不统一 | 统一为 5 维度中文名称 | 格式正确率 78%→98% |
| V1 | 2026-04-11 | Band 系统性偏高 0.5 | 增加校准规则 | 误差 0.5→0.2 |

**迭代方法**：
- 周五 18:00 抽样 10 条
- 对比 AI 评分 vs 老师纠正
- 超标则发预警，分析原因后优化 Prompt

### 3. Agent Workflow 编排

**状态机设计**：
```
IDLE → AWAITING_PART1 → AWAITING_PART2 → AWAITING_PART3 → DONE → 写Notion → 发群 → IDLE
```

**异步评分架构**：
```
学生语音 → Whisper 转写 → RAG 检索 → MiniMax 评分（暂存）→ 立即发下一题
                                                        ↓
                                              Part 3 完成后 → 汇总展示
```

**为什么不用 LangGraph**：
- 场景固定（Part1→2→3）
- 状态简单（5 种状态）
- 不需要分支逻辑
- Python 类就能实现

### 4. 模型效果评测体系

**核心指标**：

| 指标 | 定义 | 目标 | 实际 |
|------|------|------|------|
| Band 误差 | \|AI Band - 老师确认 Band\| | ≤0.3 | 0.2 |

**迭代闭环**：
```

```

## 五、面试高频 Q&A

### Q1：RAG 怎么设计的？为什么不用向量数据库？

**Answer**：
RAG 的链路是：学生答题 → Whisper 转写 → 从 Notion 错题本和作业库中检索同类话题 → 上下文注入 Prompt → MiniMax 评分。

选择关键词检索而非向量数据库，原因有三：
1. **数据量不够**：当前 < 1000 条记录，关键词检索足够
2. **数据结构化**：Notion 数据已经是结构化的（错题类型、学生答案、老师纠正），不需要文本切分
3. **够用就好**：不过度设计，未来数据量上来可以升级

---

### Q2：Prompt 迭代是怎么做的？

**Answer**：
Prompt 迭代是数据驱动的：
1. **发现问题**：通过周五的周级评估发现 Band 系统性偏高 0.5
2. **分析原因**：发现是 Prompt 里缺少"有明显错误必须扣分"的校准规则
3. **制定方案**：在 System Prompt 里增加校准规则
4. **验证效果**：下一周 Band 误差降到 0.2

第二次迭代是格式问题：发现评分输出格式不一致（维度名称有时中文有时英文），统一为中文名称后格式正确率从 78% 提升到 98%。

---

### Q3：Agent Workflow 怎么设计的？

**Answer**：
用 Python 类实现了一个简单的状态机：
- 5 种状态：IDLE / Part1 / Part2 / Part3 / DONE
- 每个状态有明确的入口动作和出口动作
- 支持老师中途发送 `/纠正` 打断流程

异步评分是关键创新：录音后不等待评分结果，直接发下一题，后台并发评分，全部答完才展示。这样学生不会有等待间隙，也不会因为看到单题反馈而影响整体发挥。

---

### Q4：模型评测体系怎么设计的？

**Answer**：
核心是 4 个指标：
- **Band 误差**：量化评分准确性
- **维度准确率**：量化每个维度（语法/词汇/...）的判断质量
- **错题命中率**：量化 AI 识别错误的能力
- **格式正确率**：量化输出稳定性

每周五自动触发评估，超标就给老师发预警，这形成了一个数据驱动的优化闭环。

---

### Q5：数据飞轮怎么设计的？

**Answer**：
数据飞轮的核心是"老师纠正"这个动作：
- 每一次老师纠正 = 1 条高质量标注数据（专家审核过的"正确答案"）
- 这些数据进入错题本 → 被 RAG 检索 → 增强后续评分
- 当错题本积累到 100+ 条后，可以作为微调数据

这意味着系统越用越聪明，因为老师在持续教它。
